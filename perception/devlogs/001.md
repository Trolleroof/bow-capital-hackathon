# Devlog 001 -- Targeting Module Bootstrap
**Date:** 2026-05-30
**Author:** Matthieu Fuller (Targeting Lead)
**Module:** `perception/`

---

## What got built

Full targeting pipeline from scratch, running on CPU/CUDA, publishable to the Outcast Virus WebSocket bus.

### Pipeline overview

```
video frame
  --> YOLOv8n/YOLO11n (TensorRT on Jetson)
      --> COCO label remap to battlefield taxonomy
      --> Haar cascade face detection (troop crops only)
  --> Norfair multi-object tracker
      --> CandidateBuffer (30-frame rolling majority vote)
      --> Operator confirm gate (single confirmed target)
  --> ReID gallery (OSNet-x0.25, part-based embedding)
  --> WebSocket bus (topic: "detections")
  --> OpenCV debug overlay
```

---

## Key design decisions

### YOLO + Haar face detection
Used OpenCV's built-in Haar cascade for face detection instead of MediaPipe -- MediaPipe dropped the `solutions` API in newer versions and has platform-specific install issues. Haar runs inside person/troop bounding boxes only, adding a `has_face` flag that feeds into the priority scorer. No new dependencies.

### Battlefield label taxonomy
YOLO is trained on COCO. Rather than fine-tuning (no time, no labeled data), remapped COCO classes at inference:
- `person` -> `troop`
- `car/truck/bus` -> `vehicle`
- `motorcycle/bicycle` -> `ugv`
- `airplane/helicopter` -> `aerial`

Judges see `TROOP 0.87` not `person 0.87`. Same model, better story.

### CandidateBuffer -- stable proposed target
Without buffering, with 10+ people in frame the "PROPOSED" label flickers between targets faster than an operator can click. Fixed with a 30-frame rolling majority vote -- the displayed candidate is whichever track_id won the most frames in the last 30. A new candidate needs to dominate for ~15 frames before displacing the current proposal. Configurable via `CANDIDATE_BUFFER_FRAMES`.

### Single confirmed target
Operator can only confirm one target at a time (`_confirmed_ids = {track_id}` not `.add()`). Added `u` keybind to unconfirm. This is both the honest UX and the correct human-in-the-loop framing.

## ReID system -- cross-frame identity preservation

This was the hardest problem: confirmed target leaves frame, Norfair assigns a new track_id when they return. Without ReID the confirmed status is lost permanently.

### Why color matching doesn't work
Everyone in camo ends up in the same region of color space. Even per-region histograms (head/torso/legs separately) aren't stable enough across lighting and angle changes from a drone.

### Why face recognition doesn't work
Drone camera resolution is insufficient for the geometric precision face recognition requires.

### Solution: metric learning embeddings + part-based matching

**Backbone: OSNet-x0.25 (torchreid)**
MobileNetV3 was the first attempt but its embeddings are from ImageNet classification training -- optimized to separate object categories, not individuals. Same-person vs. different-person similarities were in the 0.1-0.4 range with heavy overlap, impossible to threshold reliably.

OSNet is trained with triplet loss on ReID datasets (Market-1501, DukeMTMC). The loss explicitly pushes same-person embeddings together and different-person embeddings apart. With OSNet: same-person typically 0.80-0.95, different-person 0.20-0.45. A workable gap.

Falls back to MobileNetV3 gracefully if torchreid isn't installed.

**Part-based embedding**
Split each crop into three horizontal strips, embed each independently, weighted average:
- Head/shoulders: 0.20
- Torso/loadout: 0.50 -- most discriminative for uniformed targets (vest, pouches, patches)
- Legs: 0.30

Output is still 512-dim L2-normalized, so all downstream similarity logic is unchanged. Falls back to global embedding for small crops (< 48px tall) or strips (< 16px).

**Passive pre-buffering**
Rather than snapshotting on confirm (cold gallery), every tracked object gets sampled at ~1s intervals into a per-track rolling deque. On confirm, that track's pre-buffer is promoted to the confirmed gallery -- so the gallery is already rich when the operator presses `c`. Other tracks' buffers are dropped immediately (not interesting).

**False positive mitigation**
Two architectural gates:
1. ReID match only runs when the confirmed target is absent from the tracker -- prevents lookalikes in the same frame from tripping the gallery.
2. Streak gate: a candidate track must exceed threshold for 3 consecutive frames before reassignment fires -- one lucky frame doesn't trigger a false positive.

## Architecture: what changed over iterations

| Version | Problem | Fix |
|---------|---------|-----|
| v1 | MediaPipe AttributeError on Windows | Replaced with OpenCV Haar cascade |
| v2 | Proposed target flickering with many people in frame | CandidateBuffer (30-frame rolling vote) |
| v2 | Multiple confirmed targets possible | Single confirm, `u` to unconfirm |
| v3 | ReID misses + false positives | Only match when target absent; streak gate; multi-snapshot gallery |
| v3 | double tracker.update() in ReID pass | Removed -- Norfair was advancing twice per frame |
| v4 | Passive pre-buffering | Gallery pre-warmed before confirm via per-track deques |
| v5 | MobileNetV3 similarity gap too small to threshold | OSNet-x0.25 (metric learning backbone) |
| v5 | Single angle gallery brittle to re-entry from new direction | Part-based embedding with torso weighting |

## Config reference

All values overridable via `.env`:

| Key | Default | Notes |
|-----|---------|-------|
| `YOLO_MODEL` | `yolo11n.pt` | `.engine` on Jetson after `export_trt.py` |
| `DEVICE` | `cuda` | `cpu` for dev |
| `YOLO_CONF` | 0.40 | Lower for more detections |
| `CANDIDATE_BUFFER_FRAMES` | 30 | ~1s at 30fps |
| `REID_THRESHOLD` | 0.88 | Cosine sim cutoff (OSNet range) |
| `REID_CONSECUTIVE` | 3 | Frames before reassignment |
| `REID_GALLERY_SIZE` | 12 | Max confirmed gallery embeddings |
| `REID_PREBUFFER_SIZE` | 8 | Max per-track pre-buffer embeddings |
| `REID_SAMPLE_INTERVAL` | 30 | Frames between ReID samples |
| `REID_PART_MIN_H` | 48 | Min crop height for part-based embedding |
| `REID_PART_WEIGHTS` | [0.20, 0.50, 0.30] | Head / torso / legs |

---

## Controls

| Key | Action |
|-----|--------|
| `c` | Confirm buffered proposed candidate |
| `u` | Unconfirm + clear ReID gallery |
| `f` | Lock follow mode on proposed candidate |
| `r` | Release follow mode |
| `q` | Quit |

## Known limitations / open items

- Face detection is Haar cascade -- accurate enough for the demo but would miss profile/tilted faces. YOLO-face would be a better upgrade if time allows.
- `REID_PART_WEIGHTS` are not learned -- hardcoded heuristic. Would benefit from tuning on actual test footage.
- Bus publisher is synchronous in the main loop. Under heavy load it could block frame processing. Worth making async if FPS drops are observed.
- TensorRT export (`export_trt.py`) has not been tested on the actual Jetson Nano yet -- verify before demo.
