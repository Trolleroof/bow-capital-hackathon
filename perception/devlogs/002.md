## Bus integration

### Payload structure fix
001 shipped `_serialize()` wrapping detections under a `"data"` key:
```json
{"topic": "detections", "data": {"t": 1234, "objects": [...]}}
```
The orchestrator's `ws_server.py` strips `"topic"` and spreads the rest with `**payload`, so the frontend received `{"topic": "detections", "data": {...}}`. The frontend read `msg.objects` (flat), which was `undefined`.

Fix: flatten the payload so objects are top-level:
```json
{"topic": "detections", "t": 1234, "objects": [...]}
```

This matches the orchestrator's internal `router.publish()` format -- one source of truth.

### Separate image WebSocket
Heavy JPEG frames (`fpv_raw`) were sharing the same WS connection as control messages (`detections`, `pose`, etc.). Under load this caused control messages to queue behind large image payloads. Split into two connections:
- Port 8000: control messages only (detections, pose, commands)
- Port 8001 (`IMAGE_WS_PORT`): image frames only (`fpv_raw`, `slam_frame`)

Frontend subscribes to `CONTROL_TOPICS` on 8000 and `IMAGE_TOPICS` on 8001.

### `is_candidate` + `is_primary` fields
001 published a single `is_target: bool` that conflated followed and confirmed targets. The frontend had no way to know which track the `CandidateBuffer` had nominated -- FOLLOW was a shot in the dark.

Now publishes three separate fields:
```json
{
  "confirmed":    false,
  "is_primary":   false,
  "is_candidate": true
}
```
Invariant: `is_candidate` is only true when neither `is_primary` nor `confirmed`. These map to distinct frontend tones: `amber` (red) for primary, `candidate` (magenta) for proposed, no tone for passive tracks.

## Frontend HUD -- canvas overlay

### Architecture change
Moved HUD rendering from Python (baked into the JPEG stream) to a React canvas overlay on the raw frame. Benefits:
- Raw footage stays raw; annotations don't burn into recorded video
- Frontend can re-render annotations without waiting for a new frame
- Eliminates the `fpv_hud` stream entirely (half the image bandwidth)

### Detection overlay
`DetectionOverlay` / `LiveFrameCanvas` draws corner-bracket targets on a canvas that sits over the raw `<img>` / canvas element:
- Corner bracket style (no circles, no scan line) matching Python HUD
- 4 tiers: `confirmed` (lime green, thick), `primary/amber` (red), `candidate` (magenta), passive (class color, thin)
- Bracket size and label bold/normal mirror the Python `visualizer.py` logic

### Color palette sync
Python `visualizer.py` uses BGR tuples. Converted each to hex for the frontend palette:

| State/Class | Python BGR | Frontend hex |
|-------------|-----------|--------------|
| confirmed | `(0, 255, 80)` | `#50ff00` |
| primary/followed | `(60, 60, 255)` | `#ff3c3c` |
| candidate | `(200, 80, 255)` | `#ff50c8` |
| troop | `(0, 220, 255)` | `#ffdc00` |
| vehicle | `(0, 160, 255)` | `#ffa000` |
| ugv | `(30, 200, 120)` | `#78c81e` |
| aerial | `(255, 240, 50)` | `#32f0ff` |

Priority rule in both places: confirmed > primary > candidate > class color.

## Operator state machine

### FOLLOW → CONFIRM → RELEASE

Button strip in `OpticView` drives a local state machine:
```
[no follow]   → FOLLOW {T-XXXX}   (picks candidate from buffer)
[followedId]  → CONFIRM TARGET  +  RELEASE
[confirmedId] → RELEASE LOCK
```

Key design: `followedId` and `confirmedId` are local React state, not derived from WS `tone`. The WS data is authoritative for which tracks exist; operator intent is authoritative for which is highlighted.

### Flicker fix: `displayDets` override
Problem: WS detections arrive at ~30fps. After clicking FOLLOW, the optimistic `tone: 'amber'` applied by `followTarget()` was overwritten ~33ms later by the next WS frame, which still had `is_candidate: true` from the backend (command round-trip ~100-200ms). Target visibly flashed red then back to magenta.

Fix: compute `displayDets` in `OpticView` that applies local operator state on top of WS data before any rendering or candidate selection:
```typescript
const displayDets = useMemo(() => t.dets.map(d => ({
  ...d,
  tone: (confirmedId === d.numericId ? ''
         : followedId === d.numericId ? 'amber'
         : d.tone) as Detection['tone'],
  confirmed: d.confirmed || confirmedId === d.numericId,
})), [t.dets, followedId, confirmedId])
```
Local state wins immediately; backend catches up within one round-trip and they converge on the same value.

### ReID release_confirm
When the operator releases a confirmed target, Norfair assigns a fresh track ID to that person on their next detection. Without intervention, the passive pre-buffer for that new ID is empty and the gallery immediately begins re-matching against it, often re-confirming within 2-3 frames.

Fix: `tracker.release()` returns the released `track_id`. If a confirmed track was dropped, `gallery.release_confirm(track_id)` is called, which demotes the gallery back into the released track's pre-buffer so the streak counter resets cleanly.

## Anti-jitter improvements

### `TRACK_INIT_DELAY=3` (Norfair `initialization_delay`)
Norfair previously surfaced tracks on their first detection hit. Single-frame ghost detections (YOLO firing on a shadow or artifact) appeared and disappeared every few seconds.

`initialization_delay=3` requires 3 consecutive matched frames before a track becomes visible. At 30fps this is 100ms -- imperceptible for real targets, lethal for ghosts. Combined with explicit `is_initializing` filter in `tracker.update()`.

### Bbox EMA smoothing (`BBOX_SMOOTH_ALPHA=0.35`)
Raw detection bboxes vary ±5-15% between adjacent frames for the same stationary target (YOLO regressor noise). This caused corner brackets to visibly jump.

Applied exponential moving average per track ID in `TargetTracker.update()`:
```python
smoothed = [alpha * raw + (1 - alpha) * prev for raw, prev in zip(raw_bbox, prev_bbox)]
```
`alpha=0.35`: each frame moves 35% toward the new detection, 65% toward the previous smoothed position. Stale entries pruned when tracks are lost. Smoothed bbox written back to `src_det.bbox` before object is returned.

## FPS improvements

### FP16 half-precision (`detector.py`)
On CUDA devices, `self.yolo.model.half()` switches inference to float16. ~40% latency reduction with no meaningful accuracy loss on detection tasks. Applied automatically when `DEVICE != "cpu"`.

### Frame-skip detection (`DETECT_EVERY`)
YOLO is the pipeline bottleneck. New `DETECT_EVERY` knob runs the detector only every Nth frame; the Norfair tracker runs every frame and interpolates positions between detections using its Kalman filter.

```
DETECT_EVERY=1  -- 30fps detection (default, maximum accuracy)
DETECT_EVERY=2  -- 15fps detection, ~1.4x throughput gain
DETECT_EVERY=3  -- 10fps detection, ~1.8x throughput gain
```
Good target for moving cameras: `DETECT_EVERY=2`. For slow/static scenes: `DETECT_EVERY=3`.

### Grayscale FPV stream (`GRAYSCALE_STREAM`)
Converting the streamed frame to grayscale before JPEG encode reduces transmitted data by ~3x (single channel vs RGB). Detection pipeline is unaffected -- runs on full color. Enabled by default (`GRAYSCALE_STREAM=1`).

### `PROC_WIDTH` rescaling
Setting processing width rescales before YOLO. YOLO inference time scales roughly with input area, so:
- 1080p → 512px wide: input area shrinks ~4.5x, inference ~3-4x faster
- 1080p → 720px wide: input area shrinks ~2.25x, inference ~1.8x faster

Currently configured at `PROC_WIDTH=720`.

### Combined `.env` for demo day
```env
YOLO_MODEL=yolov8n.pt
PROC_WIDTH=720
DETECT_EVERY=2
FPV_INTERVAL=1
FPV_QUALITY=30
GRAYSCALE_STREAM=1
```
Expected effective throughput on laptop GPU: ~20-25fps end-to-end vs ~8fps native.

## Updated bus output format

```json
{
  "topic": "detections",
  "t": 1717084800.123,
  "objects": [
    {
      "id": 7,
      "cls": "troop",
      "conf": 0.91,
      "bbox": [0.312, 0.144, 0.088, 0.221],
      "has_face": true,
      "is_primary": false,
      "is_candidate": true,
      "confirmed": false
    }
  ]
}
```
`bbox` is normalized `[x, y, w, h]` in 0-1 coordinates relative to the processing frame. Frontend multiplies by canvas `width`/`height` to get pixel coords.
